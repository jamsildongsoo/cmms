# CMMS 제품 요구사항 정의서 (PRD v2)

> 설비유지보수관리시스템(CMMS)의 목적, 기능 요구사항, 화면 정책, 공통 규칙을 정의합니다.
> 코드 수준 구현이 아닌 **개념과 규칙** 위주로 기술합니다.

---

## 1. 프로젝트 개요

### 1.1 목적
설비 유지보수 프로세스를 디지털화하여 체계적으로 관리하고, 설비 가동률 극대화 및 유지보수 비용 절감을 달성하는 CMMS 개발

### 1.2 기술 환경

| 구분 | 기술 |
|------|------|
| Backend | Java 21 + Spring Boot 3.x + JPA |
| Frontend | React 19 + TypeScript + Vite + Tailwind CSS |
| Database | PostgreSQL 15.x+ (JSONB 지원) |
| 인증 | JWT (Access Token) + Refresh Token (HttpOnly Cookie) |
| 상태관리 | Zustand |
| UI 컴포넌트 | shadcn/ui |

### 1.3 네이밍 규칙

| 대상 | 규칙 | 예시 |
|------|------|------|
| 데이터베이스 | snake_case | equipment_id, delete_mark |
| 백엔드 코드/API JSON | camelCase | equipmentId, deleteMark |
| 프론트엔드 | camelCase | equipmentId, deleteMark |

### 1.4 API 엔드포인트 체계

| 도메인 | Prefix |
|--------|--------|
| 인증 | `/api/auth` |
| 회사 | `/api/v1/companies` |
| 기준정보 | `/api/std` |
| 마스터정보 | `/api/master` |
| 트랜잭션 | `/api/tx` |
| 재고관리 | `/api/inv` |
| 결재 | `/api/approval` |
| 메모 | `/api/memo` |
| 대시보드 | `/api/v1/dashboard` |
| 시스템(파일 등) | `/api/sys` |

## 2. 인증 및 보안 정책

### 2.1 인증 방식

- **무상태(Stateless) JWT 기반** 인증
- **Access Token**: 단기 유효(12시간), 프론트엔드 메모리(Zustand)에 저장, API 호출 시 `Authorization: Bearer {token}` 헤더에 포함
- **Refresh Token**: 장기 유효(7일), HttpOnly Cookie로 전송, 서버 DB에 저장하여 회전(Rotation) 및 강제 무효화 지원
- **비밀번호**: BCrypt 단방향 해싱

### 2.2 Silent Refresh

프론트엔드 Axios 인터셉터가 401 응답을 감지하면 자동으로 Refresh Token을 이용해 새 Access Token을 발급받고, 실패했던 요청을 재시도합니다. 동시에 여러 요청이 실패한 경우 첫 번째 요청만 갱신을 수행하고 나머지는 대기 후 일괄 재시도합니다.

### 2.3 로그인 메타데이터

로그인 시 이전 접속 정보(시간, IP)를 기록하고 사용자에게 표시하여 비정상 접속을 인지할 수 있도록 합니다. 마지막 접속 플랜트 정보를 저장하여 재접속 시 동일 플랜트로 자동 진입합니다.

### 2.4 회사 격리 (Multi-Tenancy)

모든 데이터의 PK에 `company_id`가 포함되며, 서버 측에서 JWT의 회사 정보와 요청 데이터의 일치 여부를 검증하여 회사 간 데이터 접근을 차단합니다.

## 3. 공통 정책

### 3.1 삭제 정책 — Soft Delete

모든 엔티티의 삭제는 `delete_mark='Y'`로 처리하는 논리 삭제 원칙입니다. 데이터 복구 가능성과 참조 무결성을 유지합니다. 조회 시 `delete_mark='N'`인 데이터만 반환합니다.

### 3.2 감사 추적 (Audit Trail)

대부분의 엔티티는 공통 BaseEntity를 상속하여 다음 필드를 자동 기록합니다:
- **생성자/생성일**: 최초 저장 시 JWT에서 추출한 사용자 정보와 현재 시각
- **수정자/수정일**: 매 수정 시 자동 갱신

**예외**: 재고 관련 테이블(inventory_stock, inventory_history, inventory_closing)은 도메인 특성상 별도로 관리합니다.

### 3.3 상태 코드 체계 (Common Status)

시스템 전반에서 사용하는 공통 상태 코드:

| 코드 | 의미 | 수정 | 삭제 |
|------|------|:----:|:----:|
| **T** (Temporary) | 임시저장 — 작성 중 | O | O (작성자) |
| **A** (Approval) | 결재 진행 중 | X | X |
| **C** (Confirmed) | 확정/완료 (최종) | X | X |
| **R** (Rejected) | 반려 — 재기안 가능 | △ | X |
| **X** (Canceled) | 취소/폐기 | X | X |

### 3.4 일련번호 규칙

| 구성 | 설명 |
|------|------|
| 포맷 | `{Prefix}{yyyyMM}{seq 6자리}` (총 14자리) |
| Prefix | EQ(설비), MT(자재), IN(점검), WO(작업지시), WP(작업허가), FG(파일그룹) |
| 예시 | EQ202603000001, WO202603000012 |

회사별·엔티티별·월별로 별도 채번하며, 시퀀스 테이블에서 관리합니다.

### 3.5 공통 코드 체계

헤더-아이템 Master-Detail 구조로 관리하며, 주요 코드 그룹:

| 코드 그룹 | 항목 예시 | 용도 |
|-----------|-----------|------|
| EQ_TYPE | MECH(기계), ELEC(전기), INST(계장), UTIL(유틸리티) | 설비 유형 |
| MAT_TYPE | SPARE(예비품), CONS(소모품), TOOL(공구) | 자재 유형 |
| INSP_TYPE | PATROL(순찰), MEASURE(계측), PM(예방정비) | 점검 유형 |
| WO_TYPE | BM(고장), PM(예방), CM(개량), EM(긴급) | 작업 유형 |
| WP_TYPE | HOT(화기), CONF(밀폐), ELEC(전기), HIGH(고소) 등 | 허가 유형 |

### 3.6 첨부파일 관리

- 파일 그룹-파일 아이템 구조로, 하나의 엔티티에 여러 파일 첨부 가능
- 파일 업로드 시 SHA-256 체크섬 저장 (무결성 검증)
- 업로드 제한: 단일 파일 10MB

## 4. 기능별 요구사항

### 4.1 대시보드 (Dashboard)

#### 개요
시스템 접속 시 가장 먼저 노출되는 첫 화면으로, 플랜트의 유지보수 활동 성과와 일정을 한눈에 제공합니다.

#### 화면 구성

**상단 요약 카드**
- 당월 예방점검(PM) 실적: 완료/계획 건수 및 달성률(%)
- 당월 작업지시(WO) 실적: 완료/계획 건수 및 조치율(%)
- 당월 작업허가(WP) 건수: 승인 완료 건수 및 총 신청 건수

**중앙 캘린더**
- 월간 뷰에서 PM/WO/WP 탭으로 전환
- 계획(연한 색)과 완료(진한 색)를 색상으로 구분
- 이벤트 클릭 시 해당 문서의 상세 화면으로 다이렉트 라우팅

**하단 통계 차트**
- 설비별 작업지시 TOP 5: 건수/비용/소요시간 기준 선택형 바 차트
- 설비별 작업허가 TOP 5: 허가 건수 기준 바 차트

#### 주요 로직
- 플랜트 및 당월(`yyyyMM`) 기준으로 서버 사이드에서 통계 집계
- 캘린더 이벤트는 각 문서의 ID를 포함하여 클릭 시 해당 모듈로 즉시 이동

### 4.2 기준정보 관리 (Standard Information)

#### 대상
Company(회사), Plant(플랜트), Dept(부서), Person(사용자), Role(역할), Code(공통코드), Storage(창고), Bin(구역), Location(위치)

#### 화면 흐름
```
목록(List) ──클릭──► 수정 폼(Edit Form)
                     [저장] [목록] [삭제]
```
- 별도 조회(View) 화면 없이 목록에서 바로 수정 폼으로 이동
- 등록과 수정 화면은 동일한 폼 공유
- 관리자 중심 사용 화면으로 빠른 처리 지향

#### 주요 기능
- **다단계 부서 구조**: Parent-Child 자기참조 트리
- **사용자 관리**: 사번 기반 인증, 역할 연결, 비밀번호 관리
- **역할 기반 권한**: 메뉴 노출 및 기능 접근 제어
- **공통코드**: 좌(코드 헤더) - 우(코드 아이템) Master-Detail UI
- **창고 위치**: Storage → Bin → Location 3단계 계층

### 4.3 마스터정보 관리 (Master Data)

#### 대상
Equipment(설비), Inventory(자재/부품)

#### 화면 흐름
```
목록(List) ──클릭──► 조회(View) ──수정 버튼──► 수정(Edit)
                     [수정] [목록]              [저장] [목록] [삭제]
```
- 일반 사용자의 단순 조회 빈도가 높으므로 **조회와 수정을 분리**하여 오입력 방지

#### 설비 마스터 (Equipment)

| 정보 그룹 | 관리 항목 |
|-----------|-----------|
| 기준정보 | 설비번호, 설비명, 플랜트, 설치위치, 유형(코드), 부서, 담당자 |
| 제조사정보 | 메이커, 모델, 시리얼, 규격 |
| 재무정보 | 취득가, 잔존가, 감가상각법/기간, 설치일 |
| 운영정보 | 점검 대상 여부, PSM 대상 여부, 작업허가 필요 여부, 점검 주기/단위, 마지막/다음 점검일 |

- QR 코드 랜딩 페이지: QR 스캔 시 설비 상세 화면으로 직접 접근

#### 자재 마스터 (Inventory)
- 기준정보: 자재번호, 자재명, 유형, 단위, 부서
- 제조사정보: 메이커, 모델, 시리얼, 규격
- 현재고 요약: 재고 테이블과 조인하여 현재 보유 수량 표시

#### Excel 일괄 처리
- 설비/자재 모두 Excel 업로드(일괄 등록) 및 다운로드(목록 내보내기) 지원
- 업로드 결과: 성공/실패 건수 및 행별 오류 상세 제공

### 4.4 예방점검 관리 (Inspection)

#### 개요
설비의 주기적 예방점검 계획 수립 및 실적 기록

#### 화면 흐름
```
목록(List) ──클릭──► 조회(View) ──수정 버튼──► 수정(Edit)
                     [수정] [목록] [출력]       [임시저장] [확정] [상신] [목록] [삭제]
```

#### 핵심 개념

**Stage 구분**
- **PLN (계획)**: 점검 대상, 항목, 기준값, 예정일 등 계획 수립
- **ACT (실적)**: 계획의 점검 항목을 기반으로 실측값 입력

**계획-실적 관계 (1:N)**
- 하나의 연간 계획에 월별 실적을 다수 입력 가능
- 계획 없이도 불시 점검 등록 가능
- 실적 등록 시 계획의 점검 항목(항목명, 방법, 기준값 등)을 초기값으로 자동 로드

**점검 항목 구성**
- 항목명, 점검 방법, 최소/최대/기준값, 단위, 실측값

**상태 흐름**: T(임시) → A(결재중) → C(확정) 또는 팀장급이 직접 확정(T→C)

---

### 4.5 작업지시 관리 (Work Order)

#### 개요
설비 고장, 예방, 개량, 긴급 작업의 계획 수립 및 실적 관리. **비용과 시간의 계획 대비 실적 분석**이 핵심 목적입니다.

#### 화면 흐름
점검과 동일한 패턴 (목록 → 조회 → 수정)

#### 핵심 개념

**Stage 구분**
- **PLN (계획)**: 작업 내용, 예상 비용/시간, 예정일 수립 — 예산 확보 및 계약 기초 자료
- **ACT (실적)**: 실제 투입 비용/시간 기록 — 계획 대비 분석

**관리 항목**
- 비용(cost): 계획 예산 vs 실적 비용
- 시간(time): 계획 소요시간 vs 실적 소요시간
- 작업 유형: BM(고장), PM(예방), CM(개량), EM(긴급)

**상태 흐름**: T → A → C (점검과 동일)

---

### 4.6 작업허가 관리 (Work Permit)

#### 개요
위험작업 승인 프로세스 관리. 작업 유형별 안전 체크리스트를 양식화하여 관리합니다.

#### 화면 흐름
점검과 동일한 패턴 (목록 → 조회 → 수정)

#### 핵심 개념

**허가 유형 (복수 선택 가능)**
- COM: 일반 및 화기작업
- CONF: 밀폐공간 작업
- ELEC: 전기작업 (LOTO)
- DIG: 굴착작업
- HIGH: 고소작업
- HEAVY: 중량물 작업

**체크리스트 동적 렌더링**
- 사용자가 허가 유형을 선택하면 해당 유형의 체크리스트가 동적으로 표시
- 각 유형별 체크 항목은 프론트엔드 템플릿으로 관리하고, 입력 결과는 JSONB로 저장
- 유형별 양식이 다르므로 유연한 스키마리스 저장 방식 채택

**추가 관리 항목**
- 작업 장소, 시작/종료 시간
- 작업 개요, 위험 요인, 안전 대책
- 서명란: 작업책임자, 안전관리자 등의 서명 데이터
- 연관 작업지시 참조 (`order_id`)
- 상위 허가서 참조 (`parent_permit_id`) — 연관 추적

**실적**: 별도 실적 등록은 사용자 선택 사항 (계획 중심)

---

### 4.7 재고/자재 관리 (Inventory Management)

#### 개요
부품 및 자재의 위치별 재고 관리, 수불 처리, 월마감

#### 화면 구성

**재고현황**
- Storage(창고) → Bin(구역) → Location(위치) 3단계 필터로 위치별 재고 조회

**수불처리**
| 유형 | 동작 | 이력 기록 |
|------|------|----------|
| 입고(IN) | 재고 증가 | IN |
| 출고(OUT) | 재고 감소 (재고 부족 검증) | OUT |
| 이동(MOVE) | 출발지 감소 + 도착지 증가 | MOVE_OUT + MOVE_IN |
| 실사(ADJUST) | 실재고와 장부 차이 보정 | ADJUST_IN 또는 ADJUST_OUT |

- 배치 처리: 하나의 요청에 여러 자재를 일괄 처리 가능
- 참조 문서 연결: 작업지시 등과 연계하여 출고 사유 추적

**수불이력**
- 모든 수불 거래의 불변 로그 — 수정/삭제 불가
- 거래 유형, 일시, 수량, 금액, 참조 문서 기록

**월마감**
- 대상 연월 기준 입/출/이동/조정 수량·금액 집계
- 마감 확정(C) 후에는 해당 기간의 수불 이력 수정 불가

---

### 4.8 결재 관리 (Approval)

#### 개요
트랜잭션 문서(점검, 작업지시, 작업허가)의 의사결정 프로세스 관리 및 추적

#### 결재선 구조

```
기안자(00) → 결재자(01) → 합의자(02) → ... → 참조자(03)
```

| 결재 유형 | 코드 | 처리 방식 |
|-----------|------|-----------|
| 기안 | 00 | 자동 승인 (결재선 첫 번째) |
| 결재 | 01 | 수동 — 승인 또는 반려 판단 |
| 합의 | 02 | 수동 — 승인 또는 반려 판단 |
| 참조/통보 | 03 | 자동 건너뜀 (알림 용도) |

#### 결재 결과 코드

| 코드 | 의미 |
|------|------|
| P (Pending) | 미결/대기 |
| Y (Approved) | 승인 |
| N (Rejected) | 반려 |

#### 처리 규칙

1. **상신 시**: 기안자는 자동 승인 처리, 연속된 참조 스텝은 자동 건너뜀
2. **승인 시**: 현재 스텝 승인 → 다음 스텝으로 이동. 마지막 스텝까지 승인되면 결재 완결(C)
3. **반려 시**: 즉시 결재 반려(R) 처리
4. **원문서 연동**: 결재 완결/반려 시 이벤트를 통해 원문서(점검, 작업지시 등)의 상태를 자동 반영
5. **별도 기안**: 트랜잭션 문서 외에도 독립적인 기안 작성 가능

#### 화면 구성

**상신함 (Outbox)**
- 내가 기안한 문서 목록
- 상태별 필터: T(임시), A(결재중), C(완결), R(반려)

**결재함 (Inbox)**
- 내가 처리할/처리한 문서 목록
- 구분별 필터: 미결(pending), 기결(completed), 참조(reference)

**결재 상세**
- 원문서 내용 표시
- 결재선 진행 현황 (단계별 결재자, 결과, 의견, 처리일시)
- 본인 차례일 때만 결재/반려 버튼 및 의견 입력란 활성화

### 4.9 TM 메모 (Memo / Notice)

#### 개요
당일 운영 현황 및 발생 문제를 부서 내 전파하기 위한 운영 메모 (자유게시판이 아님)

#### 핵심 정책

| 규칙 | 설명 |
|------|------|
| **수정 불가** | 확정(C) 후에는 작성자도 수정/삭제 불가 — 신뢰성 있는 현황 기록 |
| **임시저장** | T 상태에서만 수정/삭제 가능 |
| **변경 대응** | 내용 변경이 필요하면 추가 댓글 또는 신규 메모로 작성 |
| **댓글** | 1단계 댓글 — 추가 정보 및 조치 현황 기록, 본인 댓글만 삭제 가능 |
| **공지** | 공지 설정 시 목록 상단 고정 및 강조 표시 |
| **삭제 제한** | 원칙적 불가, 관리자에 한해 제한적 허용 |

#### 화면 흐름
```
목록(List) ──클릭──► 상세(Detail)      ← 확정 메모: 읽기 전용 + 댓글
            ──등록──► 등록(Register)    ← [임시저장] [확정] [목록]
```

## 5. 공통 UI 정책

### 5.1 레이아웃 구조

**헤더**
- 로고 + 회사 정보 + 사용자명
- 플랜트 전환 드롭다운 (변경 시 전역 컨텍스트 업데이트)
- 알림 아이콘 (결재 대기 건수, 점검 임박 건수)
- 프로필/로그아웃

**사이드바**
- 아코디언 메뉴 구조
- 현재 경로 자동 하이라이트 및 해당 메뉴 자동 펼침
- 역할(role) 기반 메뉴 노출 필터링
- 모바일: 오버레이, 데스크탑: 고정

**콘텐츠 영역**
- Error Boundary로 감싸서 런타임 에러 시 복구 UI 제공
- 전역 로딩: 모든 API 호출 시 자동 로딩 오버레이 표시

### 5.2 메뉴 구조

| 대메뉴 | 소메뉴 | 화면 흐름 |
|--------|--------|-----------|
| 대시보드 | 종합 현황 | 단일 화면 |
| TM 메모 | 메모 관리 | 목록 → 등록/상세 |
| 기준정보 | 회사/플랜트/부서/사용자/역할/코드/창고 | 목록 → 수정 폼 |
| 마스터정보 | 설비/자재 마스터 | 목록 → 조회 → 수정 |
| 예방점검 | 점검 계획/실적 | 목록 → 조회 → 수정 |
| 작업지시 | 작업 계획/실적 | 목록 → 조회 → 수정 |
| 작업허가 | 작업허가 신청 | 목록 → 조회 → 수정 |
| 재고관리 | 재고현황/수불처리/수불이력 | 목록/상세 |
| 결재관리 | 기안작성/상신함/결재함 | 목록 → 상세 |

### 5.3 화면 흐름 패턴별 버튼 정책

#### A. 기준정보 패턴 (목록 → 수정)
- 수정 폼: [저장] [목록] [삭제]

#### B. 마스터정보 패턴 (목록 → 조회 → 수정)
- 조회: [수정] [목록]
- 등록: [저장] [목록]
- 수정: [저장] [목록] [삭제]

#### C. 트랜잭션 패턴 (목록 → 조회 → 수정)
- 조회: [수정] [목록] [출력]
- 등록: [임시저장(T)] [확정(C)] [상신(A)] [목록]
- 수정: [임시저장(T)] [확정(C)] [상신(A)] [목록] [삭제]

**버튼 동작 규칙**:
- **임시저장**: 필수값 체크 완화, 작성 중 저장 허용
- **확정**: 팀장급 권한, 결재 없이 즉시 완료 처리
- **상신**: 결재선 지정 후 결재 요청, 이후 수정 불가
- **삭제**: 작성자 본인 + status=T일 때만 표시
- **출력**: 인쇄 전용 화면으로 이동

### 5.4 반응형 디자인
- 모바일 우선(Mobile First) 접근
- 기본 1열 → 중간 2열 → 대형 4열 그리드 레이아웃
- 라벨:입력 필드 비율 1:3 유지
- 인쇄 시 불필요한 요소 자동 숨김

### 5.5 공통 UI 컴포넌트

| 컴포넌트 | 용도 |
|----------|------|
| 검색 가능 드롭다운 | 부서, 담당자, 설비 등 대량 데이터 선택 |
| 파일 첨부 | 드래그앤드롭 업로드, 파일 그룹 기반 |
| 상태 뱃지 | T/A/C/R/X 상태별 색상 구분 |
| 리치텍스트 에디터 | 서식 편집, 테이블 삽입 (결재 본문 등) |
| Excel 업로드 모달 | 템플릿 다운로드, 업로드 결과 표시 |
| 인쇄 컴포넌트 | 점검/작업지시/작업허가/결재용 인쇄 레이아웃 |

### 5.6 데이터 표시 규칙
- 모든 폼 하단에 감사 정보(생성자, 생성일, 수정자, 수정일) 읽기 전용 표시
- 목록에서 삭제된 데이터는 표시하지 않음
- 날짜: `yyyy-MM-dd`, 일시: `yyyy-MM-dd HH:mm`

## 6. 비기능 요구사항

### 6.1 보안
- 역할 기반 접근 제어(RBAC)
- 비밀번호 BCrypt 단방향 해싱
- JWT 기반 무상태 인증
- Refresh Token DB 저장 및 사용 시 회전(Rotation)
- 파일 무결성 검증 (SHA-256 체크섬)

### 6.2 데이터 무결성
- Soft Delete로 데이터 영구 보존
- 감사 필드로 모든 변경 추적
- 회사별 데이터 완전 격리
- 재고 수불 이력 불변 로그

### 6.3 사용성
- 전역 로딩으로 API 처리 상태 시각적 피드백
- Error Boundary로 비정상 상태 복구 지원
- Silent Refresh로 사용자 세션 끊김 최소화
- 반응형 디자인으로 다양한 디바이스 지원
