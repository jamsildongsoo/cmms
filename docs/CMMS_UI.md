# CMMS(유지보수 관리 시스템) UI 설계서

본 문서는 리파워에너지 CMMS의 화면 구성, 메뉴 구조 및 주요 프로세스(설비, 예방점검, 안전작업허가 등)의 UI 설계를 정의합니다.

## 1. 공통 레이아웃 구성

### 1.1 로그인 화면
*   **중앙 정렬**: 회사 로고, ID(사번), PW 입력 필드.
*   **기능**: 로그인 상태 유지, 비밀번호 초기화 요청.

### 1.2 로그인 이후 메인 레이아웃
*   **헤더(Header)**:
    *   **플랜트 선택**: 사용자가 권한을 가진 플랜트(사업장) 전환 드롭다운.
    *   **알림 센터**: 결재 요청, 점검 알람(D-Day) 아이콘.
    *   **사용자 정보**: 소속 부서, 성명, 로그아웃 버튼.
*   **좌측 메뉴(Sidebar)**: 아코디언 형태의 대메뉴/소메뉴 구성.
*   **메인 영역(Main Area)**: 상단 Breadcrumb(홈 > 대메뉴 > 소메뉴), 본문 영역.

## 3. 주요 화면 구성 및 흐름

### 3.1 공통 화면 패턴
*   **목록(List)**: 검색 조건(상단) + 데이터 그리드(중앙) + 페이징(하단).
*   **등록/수정(Form)**:
    *   **헤더**: 타이틀 + 버튼 그룹 (저장(임시)/확정/상신/삭제/목록).
    *   **본문**: 논리적 그룹으로 나뉜 입력 폼 (기본정보, 상세정보, 첨부파일 등).
*   **조회(View)**: 읽기 전용 폼. [수정] 버튼 클릭 시 수정 모드로 전환.

### 3.2 상세 화면 설계


#### **3.2.1 기준정보 (STANDARD_INFO)**
*   **회사(Company), 공장(Plant), 부서(Dept), 사용자(User)**:
    *   **화면 구성**: 목록(List) + 등록/수정(Modal Form).
    *   **컬럼**: [PK]ID, 명칭, 상위ID, 설명, 사용여부 등.

*   **공통코드 (Common Code)**:
    *   **화면 구성**: **Master-Detail (좌우 분할)**.
    *   **좌측 (Code Header)**: 코드 그룹 목록. (예: `EQ_TYPE`)
        *   **컬럼**: 코드ID, 코드명, 설명, 사용여부.
    *   **우측 (Code Item)**: 선택된 그룹의 상세 코드 목록. (예: `COMPRESSOR`, `PUMP`)
        *   **컬럼**: 상세코드ID, 상세코드명, 정렬순서, 사용여부.
    *   **기능**: 그룹 선택 시 우측 목록 갱신. 각 영역별 [추가], [저장], [삭제] 버튼.

#### **3.2.2 마스터 정보 (MASTER_DATA)**




**A. 설비 마스터 (EQUIPMENT)**
*   **화면 구성**: **목록(List) + 상세(Form)** (화면 전환 또는 모달).
*   **목록 (List)**:
    *   **검색**: 설비명, 설비코드, 관리부서, 설치위치, 설비유형.
    *   **그리드**: 설비코드, 설비명, 설비유형, 관리부서, 설치위치, 상태, 모델명.
*   **상세 (Form)**:
    *   **레이아웃**: **4 Column 그리드** (기본 정보 위주).
    *   **섹션 1: 기본정보 (Basic Info)**
        *   **Row 1**: [PK]설비코드(1) / 설비명(3) 
        *   **Row 2**: 설비유형(code_item)(1) / 관리부서(dept_id)(1) / 설치위치(2)
    *   **섹션 2: 제원정보 (Specification)**
        *   **Row 3**: 제조사(maker)(1) / 모델명(model)(1) / 규격(spec)(1) / 시리얼번호(serial)(1)
    *   **섹션 3: 재무정보 (Financial)**
        *   **Row 4**: 설치일자(1) / 취득금액(1) / 잔존가액(1) / 감가상각방법(1) /내용연수(1)
    *   **섹션 4: 보전정보 (Maintenance)**
        *   **Row 5**: PSM대상여부(check)(1) / 예방점검여부(check)(1) / 작업허가여부(check)(1)
        *   **Row 6**: 점검주기(1) / 마지막 점검일(1) / 다음 점검일(1)
    *   **섹션 5: 기타 (Etc)**
        *   **Row 7**: 비고(note)(4) / 첨부파일(file)(4)

**B. 자재 마스터 (INVENTORY)**
*   **화면 구성**: **목록(List) + 상세(Form)**.
*   **목록 (List)**:
    *   **검색**: 자재명, 자재코드, 규격, 자재유형.
    *   **그리드**: 자재코드, 자재명, 규격, 단위, 자재유형, 제조사, 현재고.
*   **상세 (Form)**:
    *   **레이아웃**: **4 Column 그리드**.
    *   **섹션 1: 기본정보 (Basic Info)**
        *   **Row 1**: [PK]자재코드(1) / 자재명(3) 
        *   **Row 2**: 자재유형(code_item)(1) / 기본단위(unit)(1) / 관리부서(1) / 창고위치(1)
    *   **섹션 2: 제원정보 (Specification)**
        *   **Row 3**: 제조사(maker)(1) / 모델명(model)(1) / 규격(spec)(2)
    *   **섹션 3: 관리정보 (Management)**
        *   **Row 4**: 안전재고량(1) / 최대재고량(1) / 발주리드타임(1) / 단가(1)
    *   **섹션 4: 기타 (Etc)**
        *   **Row 5**: 비고(note)(4) / 첨부파일(file)(4)


#### **3.2.3 예방점검 (INSPECTION)**
*   **통합 화면 (계획/실적 공통)**: `stage` 컬럼으로 PLN(계획) / ACT(실적) 구분.
*   **공통 레이아웃**: **4 Column 그리드**.
*   **기본정보 (Header)**:
    *   **Row 1**: [PK]점검번호(1) / 점검명(2) / 날짜(date)(1)
    *   **Row 1-1(실적)**: 단계(stage - PLN/ACT)(1) / 참조 Entity(1) / 참조ID(ref_id - 점검/신청 등)(1)
    *   **Row 2**: 설비명(plant/equipment_id)(1) / 점검유형(code_item)(1) / 부서코드(dept_id) / 담당자(person_id)(1) / 
    *   **Row 3**: 비고(note)(4).
*   **점검항목 (Detail - Grid)**:
    *   **컬럼**: 순번(line_no), 항목명(name), 점검방법(method), 기준값(std)-"min,max는 database는 있으나 frontend에 표시 안 함", 단위(unit), **결과값(result_val)**.
    *   *Plan 단계에서는 결과값 비활성화 / Actual 단계에서 입력 가능.*
    *   *Status는 hidden으로 관리*

#### **3.2.4 작업지시 (WORK_ORDER)**
*   **통합 화면 (계획/실적 공통)**: `stage` 컬럼으로 구분.
*   **공통 레이아웃**: **4 Column 그리드**.
*   **기본정보 (Header)**:
    *   **Row 1**: [PK]지시번호(1) / 작업명(2) / 날짜(date)(1)
    *   **Row 1-1(실적)**: 단계(stage - PLN/ACT)(1) / 참조 Entity(1) / 참조ID(ref_id - 점검/신청 등)(1)
    *   **Row 2**: 설비명(equipment_id)(1) / 작업유형(code_item)(1) / 부서코드(dept_id) / 담당자(person_id)(1) / 
    *   **Row 3**: 비용(cost)(1) / 시간(time - ManDay)(1) 
    *   **Row 4**: 비고(note)(4).
*   **작업상세 (Detail - Grid)**:
    *   **컬럼**: 순번(line_no), 작업명(name), 작업방법(method), **조치결과(result)**.
    *   *Plan 단계에서는 작업명/방법 위주 / Actual 단계에서 결과 상세 입력.*
    *   *Status는 hidden으로 관리*
#### **3.2.5 작업허가 (WORK_PERMIT)**
*   **통합 화면 (신청/승인/완료)**: `status` 및 결재 진행 상태로 관리.
*   **공통 레이아웃**: **4 Column 그리드**.
*   **기본정보 (Header)**:
    *   **Row 1**: [PK]허가번호(1) / 작업명(name)(2) / 날짜(date)(1)
    *   **Row 1-1(실적)**: 단계(stage - PLN/ACT)(1) / 참조 Entity(1) / 참조ID(ref_id - 점검/신청 등)(1)
    *   **Row 2**: 비명(equipment_id)(1) / 부서코드(dept_id) / 담당자(person_id)(1) / 
    *   **Row 3**: 시작일시(start_dt)(1) / 종료일시(end_dt)(1) / 작업장소(location)(2)
    *   **Row 4**: 비고(note)(4).
*   **상세정보 (Form Sections)**:
    1.  **위험성평가**: 작업개요(work_summary), 위험요인(hazard_factor), 안전조치(safety_factor).
    2.  **특별 작업 (Multi-Check)**:
        *   **항목**: [ ]밀폐 [ ]화기 [ ]전기 [ ]고소 [ ]굴착 등 (복수 선택 가능 -> `wp_types` 저장).
        *   **상세 입력**: 선택한 항목에 따라 동적으로 Form 생성 (예: 밀폐 선택 시 가스측정 입력란 표시).
        *   **저장**: 모든 상세 항목은 각각의 **JSON (`checksheet_json_hot`)** 객체 내에 저장.
    *   **서명은 시스템에서 처리하지 않고 출력 폼에 넣어서 처리**
    *   **출력 폼은 일반과 특별작업은 각자 분리해서 출력**

#### **3.2.6 재고관리 (INVENTORY_STOCK)**
*   **재고 처리**:
    *   **list형태로**: 재고번호, 유형(입,출,이,조), 수량, 위치(storage/bin/loc)
*   **재고 현황 및 수불 (STOCK & HISTORY)**:
    *   **위치 관리**: 창고(storage_id) - 구역(bin_id) - 위치(location_id) - 자재(inventory_id) 계층 구조.
    *   **현황 (Stock)**: [PK]창고/구역/위치/자재 -> 수량(qty), 금액(amount).
    *   **이력 (History)**: 일자(tx_date), 유형(tx_type - IN/OUT/MOVE/ADJUST), 수량, 금액, 참조문서(ref_id).
    *   **마감 (Closing)**: 월별(yyyymm) 기초/입고/출고/이동/조정/기말 수량 및 금액 집계.

#### **3.2.7 결재관리 (APPROVAL)**
*   **통합 화면 (기안/결재)**:
    *   **기본정보**: [PK]결재번호, 제목(title), 내용(content), 참조문서(ref_entity/ref_id), 상태(status).
    *   **결재선 (Detail - Grid)**:
        *   **컬럼**: 순번(line_no), 결재자(person_id), 구분(decision - 결재/합의/통보), 결과(result - 승인/반려), 의견(comment), 처리일시(decided_at).
    *   **첨부파일**: file_group_id.

## 4. UI/UX 디자인 가이드라인 (Design Guidelines)

1.  **레이아웃 및 그리드 시스템 (Layout & Grid)**:
    *   **Form 그리드 (Columns per Row)**:
        *   **기본 (Standard)**: 한 줄에 **2개 컬럼 (2 Cols)** 구성을 원칙으로 함 (`grid-cols-1 md:grid-cols-2`).
            *   *예: 기본 정보, 제조사 정보 등 텍스트 위주 항목.*
        *   **고밀도 (Dense)**: 숫자, 날짜 등 짧은 데이터가 많은 경우 한 줄에 **3개 또는 4개 컬럼 (3~4 Cols)** 구성 (`md:grid-cols-3`, `md:grid-cols-4`).
            *   *예: 재무 정보(4 cols), 운영 정책(3 cols).*
        *   **모바일 (Mobile)**: 화면 폭이 좁은 경우 자동으로 **1개 컬럼 (1 Col)** 수직 배치.
    *   **필드 구성 (Composition)**:
        *   **비율 (Ratio)**: 수평 배치 시 **Label : Input = 1 : 3** 비율 준수 및 유지.
        *   **반응형**: 모바일 등 좁은 화면에서는 **Label(상단) / Input(하단)** 의 Stacked 구조로 자동 전환 (`space-y-2`).

2.  **버튼 디자인 (Buttons)**:
    *   **위치 (Placement)**:
        *   **Main Actions**: 화면 우측 상단 헤더 영역에 그룹핑 (저장, 취소 등).
        *   **Internal Actions**: 섹션 헤더 우측 또는 테이블 행 내부.
    *   **크기 (Size)**:
        *   **Default**: 높이 `40px` (`h-10`), 패딩 `px-4`. (페이지 주요 액션)
        *   **Small**: 높이 `36px` (`h-9`), 패딩 `px-3`. (테이블 내 버튼, 소형 폼)
        *   **Icon**: 높이/너비 `40px` 등 (`size="icon"`).
    *   **스타일 (Style)**:
        *   **Primary**: `bg-blue-600` (Solid Blue), Text White. (저장, 확정, 등록)
        *   **Secondary**: `variant="outline"` (Border slate-200). (취소, 목록, 조회)
        *   **Destructive**: `bg-red-500`. (삭제, 반려)
        *   **Ghost**: 배경 없음. (뒤로가기 화살표 등)

3.  **입력 필드 및 상태 (Inputs & States)**:
    *   **필수 입력 (Required)**: Label 텍스트 우측에 붉은색 `*` 마크 표시 (`text-red-500`).
    *   **Read-only**: 계획(Active) 데이터와 구분되도록 배경색 처리 또는 `disabled` 스타일 적용.
    *   **상태 배지 (Status Badge)**:
        *   **T (임시)**: `bg-gray-100` / `text-gray-800`
        *   **A (진행/결재)**: `bg-blue-100` / `text-blue-800`
        *   **C (완료/승인)**: `bg-green-100` / `text-green-800`
        *   **R (반려/장애)**: `bg-red-100` / `text-red-800`
